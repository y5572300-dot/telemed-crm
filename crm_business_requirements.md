# CRM «Проверка гипотез» — бизнес-требования

Версия 1.0 · Июль 2026
Заказчик: Product Owner (Ярослава)
Прототип интерфейса: `crm_hypothesis.html` в этом репозитории — является эталоном UI и логики карточки, требования ниже описывают целевую систему поверх него.

---

## 1. Назначение инструмента

Команда проверяет продуктовые гипотезы через серии интервью и переговоров с целевыми компаниями (CustDev). Инструмент решает две проблемы:

1. **Нет прозрачности воронки контактов** — непонятно, кто с какой компанией работает, на каком этапе контакт, у кого какая нагрузка.
2. **Нет общей картины для статус-встреч** — каждый участник ведёт контакты у себя, история переговоров не видна остальным.

Инструмент — общая доска контактов с воронкой по статусам, историей действий по каждому контакту и напоминаниями. Конечная цель воронки — проведённое целевое интервью / пройденные шаги проверки гипотезы.

Пользователи: 3–7 человек (PO, продажи, эксперты). Все видят все данные — разграничение прав не требуется.

---

## 2. Ключевые сущности

### 2.1 Проект

Верхний уровень. У компании несколько параллельных направлений проверки гипотез, у каждого — свой набор контактов.

| Поле | Тип | Обязательное |
|---|---|---|
| Название | строка | да |
| Описание | текст | нет |
| Набор гипотез проекта (справочник для карточек) | список строк | да |
| Статус | активный / архивный | да |

Примеры проектов: «B2B — корпоративный сегмент», «B2G — адаптация ветеранов».

**Требования:**
- Пользователь может создать проект, переименовать, архивировать (не удалять — данные сохраняются).
- Справочник гипотез задаётся на уровне проекта и используется в карточках этого проекта.
- Карточки контактов принадлежат ровно одному проекту.

### 2.2 Карточка контакта

Основная рабочая единица. Поля:

| Поле | Тип | Обязательное |
|---|---|---|
| Компания | строка | да |
| Ответственный | пользователь системы | да |
| Какую гипотезу проверяем | множественный выбор из справочника проекта | нет |
| Источник контакта | справочник: тёплый (личные связи) / холодный (исходящий) / входящий / конференция / партнёр | нет |
| Контакт внутри компании | текст (ФИО, должность, контекст знакомства) | нет |
| Контакты | текст (телефон, email, telegram — в свободной форме) | нет |
| Статус | см. воронку ниже | да |

### 2.3 Воронка статусов

Канбан-доска, колонки слева направо:

1. **Новый** — контакт найден, работа не начата
2. **Отправлен запрос** — написали, ждём ответа
3. **Согласовано интервью** — договорились о встрече
4. **Провели интервью** — целевое действие достигнуто
5. **Отказ** — контакт закрыт с причиной

Статус меняется вручную из карточки. Смена статуса фиксируется в истории действий автоматически (кто, когда, из какого в какой).

### 2.4 История действий (комментарии)

Хронологическая лента внутри карточки — главный контент карточки, визуально акцентный (см. прототип: тайм-лайн).

- Запись: дата (проставляется автоматически, сегодняшняя), автор (текущий пользователь), текст в свободной форме.
- Сортировка: новые сверху.
- Автор может удалить свою запись.
- Последняя запись видна на карточке в канбане (обрезанная).

### 2.5 Напоминания

Задачи с датой внутри карточки.

- Поля: дата, текст, автор.
- Напоминания с наступившей/будущей датой отображаются в общей панели над доской (отсортированы по дате, клик открывает карточку).
- Напоминание можно закрыть («выполнено») — оно исчезает из панели.
- **Желательно (фаза 2):** уведомление ответственному в Telegram/email в день срока.

## 3. Сценарии работы

### 3.1 Вход и выбор проекта (новое требование)

1. Пользователь открывает инструмент, авторизуется (см. §5).
2. **Первый экран — выбор проекта:** список активных проектов с количеством карточек и краткой сводкой (сколько в каждом статусе).
3. Выбор проекта открывает его канбан-доску. Инструмент запоминает последний открытый проект.
4. Переключение между проектами — из шапки, без выхода.

### 3.2 Добавление контакта

Кнопка «+ Добавить контакт» → форма с полями карточки → карточка появляется в колонке «Новый» текущего проекта. Поле «Ответственный» по умолчанию — текущий пользователь.

### 3.3 Ведение контакта

Пользователь открывает карточку, добавляет записи в историю после каждого касания, ставит напоминания («если не ответит до пятницы — написать на почту»), двигает статус.

### 3.4 Статус-встреча команды

Ключевой сценарий. Команда открывает доску на общем экране, по очереди фильтрует по ответственному («покажи карточки Ярославы») и рассказывает по своим карточкам. Требование: фильтр по ответственному — кнопки с именами и счётчиком карточек над доской, один клик.

### 3.5 Фильтрация и поиск

- Фильтр по ответственному (обязательно, см. 3.4)
- Фильтр по гипотезе (желательно)
- Текстовый поиск по названию компании (желательно)
- Фильтры комбинируются

---

## 4. Панель напоминаний

Горизонтальная панель над доской: ближайшие напоминания всех карточек текущего проекта (дата + компания + текст). Клик — открывает карточку. Если напоминаний нет, панель скрыта.

---

## 5. Пользователи и авторизация

- Простая авторизация: вход по корпоративному email (Google OAuth) или по приглашению-ссылке. Без сложной ролевой модели.
- Имя пользователя используется: в поле «Ответственный», как автор записей истории, в фильтре.
- Все авторизованные пользователи видят все проекты и все карточки; создавать/редактировать может любой.

---

## 6. Нефункциональные требования

| Требование | Значение |
|---|---|
| Общая база данных | изменения одного пользователя видны остальным без перезагрузки (polling ≤ 30 сек или WebSocket) |
| Одновременное редактирование | допустимо простое разрешение «последняя запись побеждает»; блокировки не нужны |
| Объём данных | сотни карточек на проект, до 10 проектов — высоконагруженная архитектура не нужна |
| Устройства | десктоп-браузер основной; мобильная адаптация — желательно (просмотр и добавление записей) |
| Хостинг | внутренняя инфраструктура компании или облако с доступом по ссылке для команды |
| Экспорт | выгрузка карточек проекта в CSV/Excel (желательно, фаза 2) |
| Резервное копирование | ежедневный бэкап базы |

## 7. Дизайн

Эталон — прототип `crm_hypothesis.html`:

- Светлая тёплая гамма (фон #f5f4f0, акцент #4a6fa5), без ярких цветов
- Канбан с цветными точками статусов
- Карточка открывается боковой панелью справа
- История действий — вертикальный тайм-лайн, визуально главный блок карточки
- Кнопка «Сохранить» всегда видна (закреплена в шапке карточки)

---

## 8. Вне рамок (не делать)

- Ролевая модель и права доступа
- Интеграции с почтой/телефонией
- Автоматические цепочки писем
- Аналитика/дашборды сверх счётчиков в колонках (фаза 2 — конверсия по этапам воронки)
- Публичный доступ без авторизации

---

## 9. Приоритет фаз

**Фаза 1 (MVP):** проекты + карточки + канбан + история + напоминания + фильтр по ответственному + общая база + авторизация.

**Фаза 2:** уведомления о напоминаниях в Telegram/email, фильтр по гипотезе, поиск, экспорт CSV, конверсия воронки, мобильная адаптация.

---

## 10. Критерий приёмки MVP

Команда из 5 человек ведёт два проекта параллельно; на статус-встрече доска открывается на общем экране, фильтруется по каждому ответственному; все изменения, внесённые с разных устройств, видны всем; данные не теряются при закрытии браузера.
